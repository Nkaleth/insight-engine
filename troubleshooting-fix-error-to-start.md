## 1. Contexto del Problema

El proyecto **Insight Engine** es un monorepo Turbo con dos aplicaciones:

| Aplicación | Stack | Propósito |
|------------|-------|-----------|
| `apps/frontend` | Next.js 16.2.4 (Turbopack) | Interfaz de usuario — Puerto 3000 |
| `apps/backend` | NestJS 11 + Prisma + BullMQ | API REST + IA — Puerto 3001 |

Al ejecutar `pnpm run dev`, el sistema de arranque presentaba múltiples fallos que impedían el correcto funcionamiento del proyecto.

---

## 2. Síntomas Observados

### 2.1 Error crítico — HTTP 404 en `/login`

```
frontend:dev:  GET /login 404 in 154ms (next.js: 61ms, application-code: 93ms)
```

Al abrir `http://localhost:3000`, el proxy de autenticación redirigía al usuario a `/login`, pero Next.js respondía con un **404** porque la ruta no existía.

### 2.2 Warning — Convención `middleware` deprecada

```
⚠ The "middleware" file convention is deprecated.
  Please use "proxy" instead.
  Learn more: https://nextjs.org/docs/messages/middleware-to-proxy
```

Next.js 16.x renombró la convención del archivo de `middleware.ts` a `proxy.ts`.

### 2.3 Warning — Workspace root ambiguo (Turbopack)

```
⚠ Warning: Next.js inferred your workspace root, but it may not be correct.
  We detected multiple lockfiles and selected the directory of
  /home/niltonsegura/Documents/Projects/insight-engine/pnpm-workspace.yaml
  as the root directory.
  Detected additional lockfiles:
    * /home/niltonsegura/Documents/Projects/insight-engine/apps/frontend/pnpm-workspace.yaml
```

Turbopack encontró un archivo `pnpm-workspace.yaml` duplicado dentro de `apps/frontend/`, generando ambigüedad sobre cuál era el workspace root real.

### 2.4 Backend no levantaba con Turbo

Turbo ejecutaba el script `dev` de cada paquete, pero el backend no tenía ningún script llamado `dev` en su `package.json`. Solo tenía `start:dev`. Turbo lo detectaba como paquete en scope, pero nunca lo iniciaba.

```
• Packages in scope: backend, frontend
• Running dev in 2 packages
```
→ Solo el frontend iniciaba. El backend permanecía silencioso.

### 2.5 URL del API apuntaba al frontend

```
# apps/frontend/.env.local (ANTES)
NEXT_PUBLIC_API_URL="http://localhost:3000/api"
```

El cliente Axios del frontend apuntaba al **mismo puerto del frontend** (`3000`). Toda llamada al backend (`/auth/login`, `/ai/analyze`) fallaba con un bucle de auto-petición.

### 2.6 Metadata desactualizada del layout

```tsx
// app/layout.tsx (ANTES)
export const metadata: Metadata = {
  title: "Create Next App",
  description: "Generated by create next app",
};
```

El título de la aplicación en el navegador mostraba el texto por defecto de `create-next-app`, sin relación con el proyecto real.

### 🆕 NUEVO — `apps/frontend/hooks/useLogin.ts`

**Tipo:** Archivo nuevo  
**Sprint relacionado:** Sprint 17 (API Client) — extensión del patrón de hooks  
**Razón:** La página de login necesitaba su propio hook de mutación para comunicarse con el endpoint `POST /auth/login` del NestJS.

**Qué hace:**
- Usa `useMutation` de `@tanstack/react-query`
- `mutationFn` hace `POST http://localhost:3001/auth/login` con `{ email, password }`
- `onSuccess`: al recibir `{ access_token }`, lo persiste como cookie `access_token` con expiración de 24 horas usando la Web API nativa (`document.cookie`)
- Después del guardado, llama `router.push("/")` + `router.refresh()` para revalidar el Server Component raíz

```typescript
export function useLogin() {
  const router = useRouter();

  return useMutation({
    mutationFn: async (credentials: LoginCredentials) => {
      const response = await apiClient.post<LoginResponse>("/auth/login", credentials);
      return response.data;
    },
    onSuccess: (data) => {
      // Cookie con 24h de vida, path=/, SameSite=Strict
      const expires = new Date();
      expires.setTime(expires.getTime() + 24 * 60 * 60 * 1000);
      document.cookie = `access_token=${data.access_token}; expires=${expires.toUTCString()}; path=/; SameSite=Strict`;

      router.push("/");
      router.refresh();
    },
  });
}
```

**Resultado:** Flujo de login funcional end-to-end ✅

### 🆕 NUEVO — `apps/frontend/proxy.ts`

**Tipo:** Archivo nuevo (renombrado desde `middleware.ts`)  
**Sprint relacionado:** Sprint 20 (Auth Middleware)  
**Razón:** Next.js 16.x deprecó la convención `middleware.ts`. El nombre correcto del archivo es ahora `proxy.ts`, y la función exportada también debe llamarse `proxy` (no `middleware`).

**Cambios respecto al original:**

| Antes (`middleware.ts`) | Después (`proxy.ts`) |
|------------------------|----------------------|
| `export async function middleware(...)` | `export async function proxy(...)` |
| Nombre de archivo: `middleware.ts` | Nombre de archivo: `proxy.ts` |

**El contenido lógico permaneció intacto:**
- Lee cookie `access_token`
- Si no existe → `redirect("/login")`
- Si existe → `jwtVerify()` con `jose` usando el `JWT_SECRET`
- Si el token es válido → `NextResponse.next()`
- Si el token es inválido/expirado → `redirect("/login")`

```typescript
// proxy.ts
export async function proxy(request: NextRequest) {
  const token = request.cookies.get("access_token")?.value;

  if (!token) {
    return NextResponse.redirect(new URL("/login", request.url));
  }

  try {
    const secret = new TextEncoder().encode(JWT_SECRET);
    await jwtVerify(token, secret);
    return NextResponse.next();
  } catch (error) {
    return NextResponse.redirect(new URL("/login", request.url));
  }
}

export const config = {
  matcher: ["/((?!api|_next/static|_next/image|favicon.ico|login).*)"],
};
```

**Resultado:** Warning de deprecación eliminado ✅

### ✏️ MODIFICADO — `apps/frontend/.env.local`

**Tipo:** Modificación  
**Razón:** La URL del API apuntaba al mismo puerto del frontend (`3000`). Toda petición HTTP del cliente Axios terminaba haciendo un loop o fallando porque Next.js no reconocía las rutas como endpoints del backend.

```diff
# .env.local
- NEXT_PUBLIC_API_URL="http://localhost:3000/api"
+ NEXT_PUBLIC_API_URL="http://localhost:3001"
  JWT_SECRET="MiSuperSecretoEnEntornoDeDesarrollo123!_Insight"
```

**Notas adicionales:**
- Se eliminó el sufijo `/api` de la URL base porque el NestJS backend **no tiene un prefijo global** configurado. Las rutas son directas: `/auth/login`, `/ai/analyze`, `/users`.
- El `apiClient` de Axios construye las URLs concatenando `baseURL + path` del hook, por ejemplo: `http://localhost:3001` + `/auth/login` = `http://localhost:3001/auth/login`

**Resultado:** Cliente Axios apunta al backend correcto ✅

### ✏️ MODIFICADO — `apps/backend/package.json`

**Tipo:** Modificación  
**Razón:** Turbopack ejecuta el script `"dev"` de cada paquete dentro del workspace. El backend NestJS **no tenía un script llamado `"dev"`** — solo tenía `"start:dev"`. Al no encontrar el script `dev`, Turbo consideraba el backend como "en scope" pero no lo iniciaba.

```diff
# apps/backend/package.json — sección "scripts"
  "build": "nest build",
  "format": "prettier --write \"src/**/*.ts\" \"test/**/*.ts\"",
+ "dev": "nest start --watch",
  "start": "nest start",
  "start:dev": "nest start --watch",
```

**Nota:** `"dev"` y `"start:dev"` ejecutan el mismo comando (`nest start --watch`). `"start:dev"` se preserva por compatibilidad con el NestJS CLI estándar.

**Resultado:** Turbo ahora inicia ambas aplicaciones en paralelo ✅

## 7. Lecciones Aprendidas

### 1. Los sprints de seguridad tienen dependencias implícitas
El **Sprint 20** (Auth Middleware) implementó la protección de rutas, pero **dependía implícitamente** de que existiera la página `/login`. Al no haber un sprint explícito para crear esa página, quedó como deuda técnica invisible.

> **Regla:** Antes de implementar un redirect a una ruta, verificar que esa ruta exista en el App Router.

### 2. La colisión de puertos es silenciosa
Cuando el backend y el frontend corren en el mismo puerto (`3000`), Next.js simplemente gana la carrera de binding. El backend falla silenciosamente sin mensaje de error claro en el output de Turbo.

> **Regla:** Definir siempre `PORT` explícitamente en el `.env` del backend cuando se usa en un monorepo.

### 3. Turbo requiere el nombre exacto del script
Turbo busca el script `"dev"` en el `package.json` de cada paquete. El NestJS CLI usa `"start:dev"` por convención, pero Turbo no lo reconoce. **Es necesario crear un alias `"dev"` explícito**.

> **Regla:** Todos los paquetes en el monorepo deben tener el script `"dev"` definido en su `package.json`.

### 4. Las deprecaciones de framework deben seguirse inmediatamente
`middleware.ts` → `proxy.ts` es un cambio de convención en Next.js 16.x. Ignorarlo no es solo un warning estético — en versiones futuras probablemente dejará de funcionar.

> **Regla:** Resolver warnings de deprecación en el mismo sprint donde se introduce la funcionalidad afectada.

### 5. Los archivos de configuración duplicados generan comportamiento no determinístico
El `pnpm-workspace.yaml` dentro de `apps/frontend/` hacía que Turbopack eligiera un workspace root de forma no determinística. Esto podría causar bugs difíciles de reproducir en entornos de CI/CD.

> **Regla:** Revisar que los archivos de configuración de workspace solo existan en la raíz del monorepo.
